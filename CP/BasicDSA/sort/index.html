    排序Sorting

    顧名思義，就是對資料排序。

    一些時間爆炸排序法

    以下默認\(n\)為資料數量

    • 每次選最大/最小的: \(O(n^2)\)

    • 若相鄰兩項前>後則swap: \(O(n^2)\)

    Can we do better? Yes, divide and conquer!

    Divide and conquer: Merge sort

    先推薦一下我寫的Divide and Conquer

    Merge Sort是這樣的

    以下假設我們要sort一個正常的(?)array，結果要是左邊小右邊大。

    Divide

    首先先把array從中間分開，然後把分裂的Array再分裂一次...，直到變成每個分裂出來的小Array只有兩個元素，這就是Divide。

    Conquer

    怎麼計算+合併答案Conquer呢?

    首先兩個元素，簡單，把左邊弄成小的，右邊大的，照樣swap是\(O(2)=O(1)\)。

    然後把兩個已經排序好，長度為2的陣列在\(O(4)\)的時間合併起來。

    我這邊寫\(O(2),O(4)\)是故意的，為了凸顯"合併需要的次數"。

    如何合併兩個已經Sort好的陣列成一個?

    這個問題其實也蠻經典的。

    要用到的是雙指針的技巧，但是這裡空間不夠了，之後會在Blog更新雙指針的介紹

    可以先參考: https://www.geeksforgeeks.org/merge-two-sorted-arrays/

    複雜度分析

    設這樣sort一個總長為\(n\)的array的時間複雜度為\(T(n)\)

    注意合併長度兩個長度為\(k\)的已經Sort好的陣列成一個需要的複雜度為\(O(2k)\)

    所以經過分析我們發現，合併的過程就是處理兩個長度一半的陣列，然後將那兩個陣列合併

    綜上，我們可以寫出遞迴式子

    \[T(n)=2T(\dfrac n 2)+O(n)\]

    複雜度分析:怎麼解遞迴

    \[T(n)=2T(\dfrac n 2)+O(n)=4T(\dfrac n 4)+2O(\dfrac{n}{2})+O(n)=...\]

    注意\(2O(\dfrac{n}{2})\)可以視為乘進去變\(O(n)\)

    \[\Rightarrow T(n)=2T(\dfrac n 2)+O(n)=4T(\dfrac n 4)+2O(n)=...\]

    然後我們一直疊代下去發現，每次\(T\)那一坨都會砍半，一個數最多能砍半多少次呢?沒錯就是\(O(\log_2 n)\)次

    所以最後那個式子會變得像這樣\(T(n)=2^{O(\log_2 n)}T(1)+O(\log_2 n)\times O(n)=O(n\log n)\)

    或是直接砸Master Theorem。

    Other good sorts

    • Quicksort也是分治的sort的一種。它的期望複雜度會是 \(O(n\log n)\) 不過會退化要小心，可能之後有機會會補上

    • Heap sort 則是引入Heap這個樹形結構來降複雜度，\(O(n\log n)\)，優點是不用額外空間，缺點是Code Hen長。

    Sorting in C++

    Sorting in C++ is really, really convenient.

    std::sort(begin,end,cmp);

    就好啦

    • 通常我們是sort數字，排序預設是從小到大

    • Begin和End都是丟iterator進去(暫時將它看成一種進階的pointer，因為這也是一門大學問，感覺好深rrrr)

    • cmp是比對函式，常用的從大到小的排序模板為

    bool cmp(int x,int y){
    return x>y;
}

    • Sort a Struct就常常需要自訂cmp函式

    • Sort的範圍是[begin,end)，要小心

    • 附上兩種最常見的用法

    //C-style array
int arr1[N];
sort(arr,arr+N,cmp);
//Using vector
vector<int> arr2(N);
sort(arr2.begin(),arr2.end(),cmp);

    番外篇: O(n)的sort???

    真的假的啦，歐恩?

    對阿，真的! Counting Sort。

    我們不訪考慮這樣: 開另一個很大的array來統計元素

    之後掃過原本的一遍，統計一下之後把統計的陣列從左到右跑一次，遇到統計\(\neq 0\)的數字就輸出幾次就好啦

    這樣得優點只要統計跑過一次，遍歷時1次，輸出一次，確實複雜度\(O(n)\)

    缺點就是顯而易見: 不是General的Sort演算法。通常是整數才能用，因為如果有小數就炸了，兩個整數間有無窮個小數根本無法開統計的陣列(有無窮項欸)，而且被Sort的資料值域不能太大，像是到\(10^{18}\)我看你怎麼統計，空間直接爆炸。
